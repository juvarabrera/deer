## High-level overview

deer is a TypeScript application that orchestrates Claude Code agents. Each agent is an independent process lifecycle:

<div class="flow-diagram">
User prompt
    │
    ▼
createWorktree()          ← new git branch + worktree on disk
    │
    ▼
startAuthProxy()          ← Node.js MITM proxy on Unix socket (host side)
    │
    ▼
launchSandbox()           ← SRT runtime wraps the command
    │
    ▼
tmux new-session          ← isolated terminal session with the SRT sandbox
    │
    ▼
claude --dangerously-skip-permissions  ← agent runs inside the sandbox
    │
    ▼
(agent works, TUI polls for status)
    │
    ▼
createPullRequest()       ← staged changes, Claude generates metadata, gh pr create
    │
    ▼
cleanupWorktree()         ← worktree removed, branch deleted from local repo
</div>

## Data layout

```
~/.local/share/deer/
  tasks/
    <taskId>/
      worktree/           ← git worktree (only writable path in sandbox)
      srt-settings.json   ← SRT sandbox settings for this agent
      deer-auth-<taskId>.sock  ← Unix socket for auth proxy
  history/
    <repohash>.jsonl      ← per-repo task history (one JSON line per task)
  auth-proxy-server.mjs   ← proxy server script (extracted from binary at first run)
  node_modules/           ← SRT package (installed by `bunx @zdavison/deer install`)
  prompt-history.json     ← TUI prompt input history
```

The `<repohash>` is a stable hash of the canonical repo path, so each repository gets its own history file.

## Agent state machine

Each agent moves through a set of well-defined states with explicit transitions:

```
setup
  │
  ├─► running ─────────► teardown ─► running (loop)
  │       │
  │       ├─► failed
  │       ├─► cancelled
  │       ├─► interrupted ─► running (on retry)
  │       └─► pr_failed
  │
  ├─► failed
  └─► cancelled
```

### States

| State | Entry condition | Exit condition |
|-------|----------------|----------------|
| `setup` | Prompt submitted | Sandbox started → `SETUP_COMPLETE` |
| `running` | Sandbox running | Claude finishes → `TEARDOWN_START` |
| `teardown` | PR teardown in progress | Complete → back to `running` |
| `failed` | `ERROR` event | Terminal — user must retry or delete |
| `cancelled` | `USER_KILL` event | Terminal — user must retry or delete |
| `interrupted` | Session closed unexpectedly | `SETUP_COMPLETE` on retry |
| `pr_failed` | `PR_FAILED` event | Terminal — user must retry PR or delete |

### Available actions per state

Each state has a base set of available actions. Additional actions (`create_pr`, `open_pr`, `update_pr`, `retry`) become available when the agent is **idle** — meaning Claude is at rest and not producing output.

Idleness is detected by comparing consecutive tmux pane captures: if the pane content is unchanged for `IDLE_THRESHOLD` (3) polls, the agent is considered idle.

## Git worktree management

When a prompt is submitted, deer calls:

```ts
const worktree = await createWorktree(repoPath, taskId, baseBranch);
```

This runs:
1. `git worktree add` to create a new directory at `~/.local/share/deer/tasks/<taskId>/worktree`
2. Creates a new branch `deer/<taskId>` from `baseBranch`
3. Configures `user.name` and `user.email` in the worktree's local git config

The worktree lives completely outside the main repo directory. Changes made by the agent do not interfere with your current working tree.

When a PR is created, the branch is renamed from `deer/<taskId>` to `deer/<generated-name>` (where the name is generated by Claude from the diff).

## Sandbox internals (SRT)

The Anthropic Sandbox Runtime (`@anthropic-ai/sandbox-runtime`) provides cross-platform process sandboxing. deer's integration works as follows:

### `srt.ts`: building the settings

```ts
// Simplified
const settings = {
  network: {
    allowedDomains: config.network.allowlist,
    mitmProxy: {
      socketPath: authProxy.socketPath,
      domains: ["api.anthropic.com"],
    },
  },
  filesystem: {
    denyRead: buildHomeDenyList(requiredPaths),
    allowWrite: [worktreePath, "~/.claude", "/tmp"],
  },
  allowPty: true,  // Claude Code needs PTY for its TUI
};
```

The settings JSON is written to `srt-settings.json` next to the worktree, and the `srt` binary is invoked with `-s srt-settings.json -c <shell-cmd>`.

### `sandbox/index.ts`: tmux session

```ts
// Simplified
Bun.spawn([
  "env", "-i",           // clear environment
  `PATH=...`, `HOME=...`, `TERM=...`,
  "tmux", "new-session", "-d", "-s", sessionName,
  "-x", "200", "-y", "50",
  "sh", "-c", preamble,  // preamble re-exports allowed env vars, then exec's the SRT command
])
```

The tmux session runs in a clean environment (`env -i`). The shell preamble re-exports exactly the variables that are allowed to reach the agent.

### Home directory deny list

At sandbox launch time, deer enumerates your `$HOME` directory and denies read access to every entry except:
- `.claude*` directories (required by Claude Code)
- Ancestors of paths that must remain readable (worktree, PATH entries, SRT binary location)

This is dynamic — new dotfiles you create in the future are automatically blocked without any configuration change.

## Auth proxy internals

The auth proxy is a Node.js HTTP server that runs on the host and listens on a Unix socket. It acts as a forward proxy for credentialed API domains.

### Request flow

```
Claude (inside sandbox)
  │  GET http://api.anthropic.com/v1/messages
  │  Host: api.anthropic.com
  │  CLAUDE_CODE_OAUTH_TOKEN: proxy-managed  (placeholder)
  ▼
SRT proxy
  │  Sees: domain "api.anthropic.com" → mitmProxy.domains
  ▼
Unix socket → auth-proxy-server.mjs (Node.js, host side)
  │  Reads: real token from upstreams[0].headers
  │  Rewrites: Authorization: Bearer sk-ant-oat01-...
  │  Forwards: HTTPS POST to https://api.anthropic.com/v1/messages
  ▼
Anthropic API (HTTPS)
```

### Why not Bun for the proxy?

Bun's Node.js compatibility layer has a known issue with long-lived streaming connections, specifically Server-Sent Events (SSE). Claude Code uses SSE for streaming responses. The auth proxy uses a real Node.js subprocess to avoid this incompatibility.

The proxy source (`auth-proxy-server.mjs`) is embedded into the compiled binary using Bun's `import ... with { type: "text" }` at compile time. At runtime, it is extracted to `~/.local/share/deer/auth-proxy-server.mjs` on first use.

## PR generation

When the user presses `p`, `createPullRequest` in `src/git/finalize.ts` does:

1. **Stage changes**: `git add -A`
2. **Find PR template**: checks standard GitHub template locations (`.github/PULL_REQUEST_TEMPLATE.md`, etc.)
3. **Generate metadata**: calls `claude -p <metadataPrompt> --output-format json` with the diff and commit log as context. Claude returns a JSON object with `branchName`, `title`, and `body`.
4. **Commit**: `git commit -m <title>` (uses generated title as commit message)
5. **Rename branch**: `git branch -m deer/<generated-name>`
6. **Push**: `git push -u origin <branch>`
7. **Create PR**: `gh pr create --title ... --body ...`

The diff is truncated at 20,000 characters before sending to Claude to avoid token limits. Only files that actually appear in the diff are referenced in the generated PR body (Claude is explicitly instructed not to hallucinate file names).

## Configuration loading

Config loading (`src/config.ts`) follows this sequence:

1. Start with `DEFAULT_CONFIG` (hardcoded in `config.ts`)
2. Read `~/.config/deer/config.toml` — deep merge on top
3. Read `<repoPath>/deer.toml` — `applyRepoLocal` (handles `_extra` suffix fields as additive merges)
4. Apply CLI overrides — deep merge on top

The `_extra` suffix convention in repo-local config allows **additive** behavior: `allowlist_extra` extends the existing allowlist rather than replacing it. This lets repos add their own registries without touching the global config.

## Contributing

```sh
git clone https://github.com/zdavison/deer
cd deer
bun install
bun test        # run the test suite
bun run dev     # run from source (requires being inside a git repo)
bun run build   # compile Linux x64 binary to dist/
```

The test suite covers:
- Agent lifecycle and state machine
- Config loading and merging
- Git worktree creation and removal
- PR finalization logic
- Sandbox auth proxy
- Security properties (API key isolation, filesystem escape prevention, network exfiltration)
